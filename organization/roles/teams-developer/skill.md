# デベロッパー スキル定義

## Legacy Notice

このスキルは、ユーザーが Claude の Teams 機能を試していた時期の旧実験用スキル。

今後の組織フローでは使わない。実作業では次を使う。

| 旧スキルの大枠 | 現行の移行先 |
|---|---|
| API、業務ロジック、サーバーサイド実装 | `tech-backend` |
| UI 実装、状態管理、画面修正 | `tech-frontend` |
| 環境、依存関係、ビルド基盤、CI 下準備 | `tech-infrastructure` |
| モバイル実装 | `tech-mobile` |
| テスト設計、再現確認、検証 | `tech-tester` |
| セキュリティ観点 | `tech-security` |
| 独立技術レビュー | `tech-reviewer` |

既存ノートにこのスキル名が出てきた場合だけ、参照用 legacy として読み替える。

## 1. 実装前の確認事項

タスクに着手する前に、以下を必ず確認する。

### 1.1 設計ドキュメントの読み込み

| 確認対象 | 確認内容 |
|---------|---------|
| 詳細設計 | 実装すべき機能・データフロー・API仕様 |
| DB設計 | スキーマ・インデックス・リレーション |
| UI/UXデザイン | コンポーネント構造・状態管理方針 |
| tech-leader の技術選定 | 採用フレームワーク・ライブラリ・バージョン |

設計ドキュメントが不明瞭または矛盾を含む場合は、**実装を開始せず** project-manager に報告して解消してから着手する。

### 1.2 既存コードの把握（Grep-first 原則）

新規ファイルを作成する前に、類似の実装が既存コードに存在しないかを必ず確認する。

```
1. Grep で関連キーワードを検索
2. 見つかったファイルを offset/limit で必要部分のみ Read
3. 再利用・拡張できるものはそのまま活用する
```

## 2. コーディング原則

### 2.1 DRY（Don't Repeat Yourself）

同じロジックが2箇所以上に出現する場合は、共通化を検討する。ただし「3の法則」を基準とする。

- 1〜2箇所: そのままでよい
- 3箇所以上: 共通化する（関数・フック・ユーティリティ・モジュール）

共通化の単位はユースケースで判断する。汎用すぎる抽象は避け、実際に使われる範囲に閉じた設計にする。

### 2.2 YAGNI（You Aren't Gonna Need It）

「将来必要になるかもしれない」機能は実装しない。現在の要件のみを実装し、過剰な汎化・拡張ポイントの追加は行わない。

拡張性が明示的に要件に含まれている場合を除き、以下は避ける：

- 使われないオプション引数
- 「念のため」のフォールバック処理
- 未定義のユースケース向けの抽象化

### 2.3 単一責任原則

1つの関数・クラス・モジュールは1つの責務のみを持つ。「〜と〜を行う」という説明が必要になったら分割を検討する。

### 2.4 変更容易性の優先

パフォーマンスより変更容易性を優先する。ただし tech-leader が設計したパフォーマンス要件がある場合はそちらを優先する。

```
優先順位:
1. 正確さ（仕様通りに動く）
2. 変更容易性（修正しやすい構造）
3. パフォーマンス（tech-leaderの設計に準拠）
```

## 3. テスト作成ガイドライン

### 3.1 テスト対象の選定基準

| テスト種別 | 対象 | 優先度 |
|-----------|------|--------|
| ユニットテスト | ビジネスロジック・ユーティリティ関数 | 高 |
| 統合テスト | API エンドポイント・DB 操作 | 高 |
| E2E テスト | クリティカルユーザーフロー | 中 |
| スナップショット | UI コンポーネント（変更検知用） | 低 |

### 3.2 テストの品質基準

良いテストの条件：

- **Arrange-Act-Assert** パターンを使う
- テスト名は「〇〇のとき〇〇すること」形式で意図を明記する
- 1テストに1アサーション（または関連するアサーションのグループ）
- テスト間で状態を共有しない（各テストは独立して実行できる）
- モックは最小限にとどめ、実際の動作に近い形でテストする

### 3.3 エッジケースのテスト

- 空値・null・undefined の入力
- 境界値（0, 最大値, 最小値）
- 非同期エラー・ネットワーク障害
- 認証エラー・権限不足

## 4. セキュリティチェック（最重要）

**コーディング完了後、project-manager に成果物を提出する前に必ず `tech-security` へレビューをhandoffする。これは省略不可の必須ステップ。Web/APIの深いレビューでは `tech-security` のbundled Web security checklistを使う。**

### 4.1 実装中に注意すべき代表的な脆弱性

| 脆弱性 | 注意点 |
|--------|--------|
| SQLインジェクション | プレースホルダ・ORMを使い、生SQLにユーザー入力を埋め込まない |
| XSS | ユーザー入力を HTML としてレンダリングしない。innerHTML を避ける |
| CSRF | 状態変更 API には CSRF トークンまたは SameSite Cookie を使う |
| 認証・認可の欠落 | APIエンドポイントごとに認証・認可チェックを実装する |
| 機密情報の漏洩 | ログ・レスポンス・クライアントバンドルに秘密情報を含めない |
| 依存関係の脆弱性 | `npm audit` / `pnpm audit` を実行し、高リスク脆弱性を解消する |

### 4.2 セキュリティレビューの手順

```
1. コーディング完了
2. tech-security へレビューをhandoff
3. 検出された脆弱性をすべて修正する
4. 再度 tech-security へhandoffして修正を確認
5. セキュリティ問題なし → project-manager に提出
```

セキュリティレビューで高優先度の脆弱性が検出された場合は、修正が完了するまで成果物を提出しない。

## 5. 成果物フォーマット

project-manager への完了報告には以下を含める。

```markdown
## 実装完了報告

### 実装内容
- **機能**: （実装した機能の概要）
- **実装ファイル**:
  - `path/to/file.ts`: （変更内容の概要）
  - `path/to/test.spec.ts`: （テスト内容の概要）

### テスト結果
- テスト件数: X件（Pass: X / Fail: 0）
- カバレッジ（重要ロジック）: 〇〇%

### セキュリティチェック結果
- 実施日時: YYYY-MM-DD
- 使用ロール: tech-security
- 検出された問題: なし / [問題がある場合は内容と対応を記載]

### 残課題・引き継ぎ事項
- （あれば記載）
```

## 6. コミュニケーションプロトコル

### 6.1 着手前の確認

- 設計に不明点・矛盾がある → project-manager または tech-leader に確認してから着手
- 設計の変更が必要と判断した場合 → 自己判断で変更せず、tech-leader に相談する

### 6.2 ブロッカーの即時報告

実装中に以下が発生した場合は即座に project-manager に報告する：

- 設計どおりの実装が技術的に不可能
- 外部依存（API・サービス）に問題がある
- セキュリティ上の問題が設計レベルで存在する

### 6.3 完了報告のタイミング

- セキュリティチェック完了後に報告する（チェック前の中間報告は行わない）
- テストが全件 Pass していることを確認してから報告する
