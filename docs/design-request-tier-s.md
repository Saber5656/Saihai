## 2. 依頼内容

以下の Tier S 4 プロダクト(開発規模: 極小 × おもしろさ: 高 × 非開発者向け、
という基準で最優先に選定済み)について、**実装に入る前の詳細設計**を
お願いします。コードはまだ書かず、設計書の作成がゴールです。

| ID | 名称 | 一言 |
|---|---|---|
| P93 | cursorpets | カーソルを追いかける古典 oneko の現代版 |
| P92 | notifybird | 通知を鳥が咥えて届ける演出レイヤー |
| P91 | keycat | Enter を叩くと画面下から猫が出てきて一緒に叩いて祝うデスクトップ演出 |
| P83 | 8bitme | 家族写真をドット絵・レトロゲーム風アバターに変換する |

技術的に重ならない 4 つなので、並行して設計を進めてもらって構いません。

## 3. 各プロダクトの設計に含めてほしい項目

1. **MVP スコープ** — v1 に入れる機能/入れない機能の線引き
2. **対応プラットフォーム** — macOS / Windows / Linux のどこまで対応するか、
   優先順位と理由
3. **技術選定** — 言語・フレームワーク・主要ライブラリとその根拠
4. **アーキテクチャ** — 主要コンポーネントと処理フロー
   (例: keycat なら「グローバルキーフック→状態管理→透過オーバーレイ描画」)
5. **UI/UX** — 主要画面、設定項目、初回起動時の体験
6. **プライバシー/安全性設計** — 特に keycat(キー入力を監視)と
   notifybird(通知内容に触れる)は、「何を見て何を見ないか」を
   アーキテクチャレベルで保証する設計にすること
   (信頼性がこの手のツールの生死を分ける)
7. **配布形態** — インストール方法(brew / npm / 単体バイナリ配布など)、
   自動更新の要否
8. **README 構成案** — バナー・デモ GIF の入れどころ・1 行セットアップまで
   3 スクロール以内、という共通プレイブックの方針に沿った骨子
9. **リスク・要検証事項** — 技術的に詰めが甘い箇所、先に触って
   確認すべきこと(例: OS ごとのキーフック API の制限事項)

## 4. 出力形式

プロダクトごとに設計書を 1 ファイルにまとめてください。
対象プロダクトの専用リポジトリが既に存在してアクセスできる場合はそこに、
なければひとまず `docs/designs/<product-name>.md` としてこのリポジトリに
置いてください。

## 5. やらないでほしいこと

- 実装コードを書くこと(このタスクは設計のみ)
- Tier S の「小さく作って早く出す」という方針を超える壮大な将来構想を
  最初から盛り込むこと(拡張案は「将来検討」として別枠に書く分には OK)
