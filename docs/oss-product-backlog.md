## オリジナル案の詳細構想

### P23 scanmd — あらゆるテキストの取り込み口

**原案(発案者メモ):** ファイル読み込みは PDF, Word, 画像と拡張子がさまざまで、
ダウンロード→エージェントに貼り付けが面倒。システム起動して画面スキャン、
カメラ起動して読み取り、のようにあらゆる形態のテキストを md に変換したい。

- **入力経路(独自価値):** ①ファイル(PDF/Office/画像/EPUB/HTML/.eml)
  ②画面スキャン(ホットキー→範囲選択→OCR、スクロールキャプチャ対応)
  ③カメラ(書類自動検出・台形補正・影除去、スマホ連携)
  ④クリップボード監視 ⑤URL 本文抽出
- **出力:** クリーンな Markdown(表→md 表、数式→LaTeX、コード→フェンス、
  frontmatter に出典・日時・ハッシュ)。宛先はクリップボード/ファイル/stdout/
  **MCP サーバー経由でエージェント直行**。
- **形態:** CLI+常駐ホットキー+MCP サーバーの三形態。
- **既存との関係:** 変換エンジンは markitdown(MS)/docling(IBM)/pandoc/
  ローカル VLM を下に敷き、**キャプチャ UX+ルーティング層**に徹する。
  既存はすべて「ファイルを渡してもらう」前提で、取り込み口を持たない。ここが空席。
- **星系:** P44 formfill / P45 papertrail / P46 whatsthis / P65 inkstory / P73 meishi の土台。

### P74 tokenhp — トークン残量の体力ゲージ化

**原案(発案者メモ):** モンハンの体力ゲージみたいにトークン使用料をチェックして、
減ったらダメージのエフェクトを画面に反映。3 割を切ったらゲージが赤くなる。

- **コンセプト:** コストの「数字」を「感情」に変換する。予算に対する残量を
  体力ゲージとして常時表示し、消費イベントで画面端に赤いダメージビネット、
  残 30% でゲージ赤化+警告音、残 0 で「力尽きた」演出。予算追加=回復薬。
- **表示形態:** ①デスクトップオーバーレイ ②メニューバー/トレイ
  ③ターミナル statusline(starship/tmux 部品)④**OBS 配信オーバーレイ** —
  ライブコーディング配信者が画面に載せる=視聴者経由で自己拡散する広告塔。
- **実装:** tokenmeter(P06)のコアを共有するフロントエンド。Claude Code /
  Codex の hooks・transcript からリアルタイム消費を取得。演出強度は設定可能
  (真面目モード=ただの残量バー、にも倒せる)。
- **なぜバズる:** コスト可視化ツールは「数字を見せる」ものばかりで、
  ゲーミフィケーションは完全に空席。クリップ映えが極端に良く、動画・配信文化と
  直結する。「AI にお財布を削られる感覚」という 2026 年の共通体験を演出に変える。
- **注意:** 特定ゲームの名称・素材・意匠は使わない(汎用の「体力ゲージ」表現で)。
  名称候補: tokenhp / manabar(トークン=MP 説)。

### P75 iotify — なんでもかんでも後付け IoT 化

**原案(発案者メモ):** なんでもかんでも IoT 化しちゃうプロダクト。

- **コンセプト:** 「配線しない・分解しない・元の機械を傷つけない」後付け IoT 化キット。
  1. **カメラ方式(主役):** 古スマホ / ESP32-CAM を貼るだけで、アナログメーター
     読み取り、洗濯機の LED 状態、ポストの投函、湯沸かし、植物の状態などを
     画像認識で「状態」に変換。ゼロ配線が最大の売り。
  2. **センサーレシピ集:** 振動(洗濯終了)、電流クランプ(家電 on/off)、
     リードスイッチ(ドア開閉)、重量(米びつ残量)等をワンクリック書き込みで。
  3. **出力:** MQTT / Home Assistant / Webhook / hearthboard(P20)・doorlog(P55)通知。
- **既存との関係:** ESPHome(ファームウェア基盤)、Home Assistant(ハブ)、
  AI-on-the-edge-device(メーター読み取りで実績)が先行。iotify は
  **「レシピカタログ+カメラ第一+ワンクリック導入」の UX 層**としてその上に乗る。
  現状これらは「電子工作ができる人」専用で、そこの民主化が空席。
- **なぜバズる:** 「実家の◯◯を IoT 化した」記事は毎回バズる鉄板ジャンル。
  レシピがコミュニティ貢献の受け皿になり(新レシピ=新コンテンツ)、改廃が賑わう。
  P57 openintercom と「古スマホ再生」基盤を共有。
- **名称注意:** iotify は既存サービス名と衝突の可能性あり。要調査(候補: retrofit-iot / mono-iot)。

### P76 chronomap — 現在地から過去へ潜るタイムトラベル地図

**原案(発案者メモ):** 地図アプリで、自分が今いる現在地から「過去ここで何が
あったか」の歴史を遡れるアプリ。

- **コンセプト:** 地図に**時間スライダー**を付ける。「いまここ」ボタン→スライダーを
  引くと、立っている場所の過去が層で現れる:年代別空中写真・旧版地図
  (国土地理院タイル)、その場所で起きた出来事(Wikidata/Wikipedia の geo 情報)、
  昔の写真(自治体デジタルアーカイブ+ユーザー投稿)、旧町名・旧河道・埋立の変遷。
- **体験:** 散歩モード(歩くと「ここは 80 年前、川だった」と通知)、
  ビフォーアフターのスワイプ比較、発見のシェアカード生成。
- **データとコミュニティ:** 国土地理院の年代別空中写真・地理院タイル、Wikidata、
  オープンな郷土資料を基盤に、**郷土史家・地元民がレイヤーと注釈を投稿できる**
  構造にする。歴史クラスタは「書きたい人」が多く、コミュニティが自走する。
- **既存との関係:** 今昔マップ on the web(タイル比較・学術)、こちずぶらり等が
  先行するが、「時間スライダー×出来事×投稿×散歩通知」を統合した OSS は不在。
  vol.3 #90 historywalk は本案に吸収。
- **なぜバズる:** 「自分の家、100 年前は海だった」系の発見は SNS で無限に拡散する。
  旧河道・埋立地は防災文脈(地盤・水害リスク)で公共性もあり、報道と自治体が
  増幅装置になる。

### P91 keycat — Enter を一緒に叩いてくれる猫

**原案(発案者メモ):** Enter キーを入力すると画面下から Bongo Cat が出てきて、
Enter を叩くように手を振ってくれるアプリ。

- **コンセプト:** 開発の一番小さな達成(Enter)を、一番小さな祝福で返す。
  普段は隠れていて、Enter で画面下からにゅっと登場し、入力に同期して手を叩く。
- **演出:** 連打でコンボ(観客が増える)、コマンド成功(exit 0)で大喝采、
  失敗時は励まし、深夜は眠そうな動き。演出強度は調整可(真面目モードあり)。
- **技術コア(プロンプト不要の本体):** グローバルキーフック(macOS CGEventTap /
  Windows LL hook / Linux evdev+layer-shell)、クリックスルー透過オーバーレイ、
  低遅延スプライト描画、**OBS ソース対応**(配信文化=拡散装置)。
- **プライバシー設計:** 監視するのは特定キー(Enter 等)のみ・内容は読まない・
  ログを残さない、をアーキテクチャで保証(キーロガー疑念の遮断が信頼の要)。
- **スキン生態系:** キャラクターはスキン形式でコミュニティ投稿可能
  (pixelforge 系と接続)。**注意: Bongo Cat は原作者のいるミームのため、
  同梱キャラはオリジナル猫とし、既存キャラは各自インポートの形にする。**
- **星系:** tokenhp(P74)・pixelagent 系と同じ「開発演出スイート」。
  8bitme(P83)でアバターを作って keycat に住ませる導線。
