# OSS プロダクト提案書 — バズる OSS を作るための 10 案

skydoves 氏([github.com/skydoves](https://github.com/skydoves))のような
「えぐいスター数」の OSS ポートフォリオを目指すための、プロダクト提案 10 件。

前提として、この提案は次の 2 つを踏まえてチューニングしてある。

1. **skydoves 氏の成功パターンの分析**(下記)
2. **あなた自身の強み** — Saihai(Agent-Teams-Viewer)で既に持っている
   「マルチエージェント組織の typed state 管理・可視化・ワークフロー契約」という資産。
   2026 年現在、エージェントインフラは CI/CD 並みの基盤技術になりつつあり、
   この領域はスター獲得の最大の追い風が吹いている。

## 0. skydoves 氏の成功の方程式

| # | パターン | 具体例 |
|---|---|---|
| 1 | **一点突破の小さなライブラリ** — 誰もが週に一度は踏む痛みを 1 つだけ解決する | Balloon(ツールチップだけ、4k★)、Sandwich(API レスポンスの型付けだけ、1.8k★) |
| 2 | **プラットフォームの波に最速で乗る** — 新技術の登場直後、まだ定番が無い時期にその技術「専用」ライブラリを出す | Jetpack Compose 登場直後の landscapist(画像読み込み、2.6k★) |
| 3 | **ショーケースリポジトリ** — ライブラリではなく「最新ベストプラクティスの完全実装例」を公開する | Pokedex(8.3k★)、chatgpt-android。実装例はライブラリよりスターが伸びやすい |
| 4 | **README が製品** — GIF デモ・バッジ・1 行セットアップ・美しいバナーを必ず揃える | 全リポジトリ共通。README を見た 10 秒で価値が伝わる |
| 5 | **量産と相互送客** — 小さなリポジトリを多数持ち、README 同士でリンクし合う | 数十リポジトリが互いにトラフィックを回す |
| 6 | **自分の実務からの切り出し** — 本業で作った部品を汎用化して公開する | Stream / RevenueCat での実務 → ライブラリ化 |

このうち特に効くのは **2(波に乗る)× 6(実務から切り出す)** の掛け算。
あなたの場合、実務 = Saihai で作っているエージェント組織基盤なので、
以下の 10 案の多くは「Saihai から切り出す・汎用化する」形になっている。

## 1. Saihai の国際化・汎用化 — 「コーディングエージェントの管制塔」

**一言ピッチ:** `k9s` が Kubernetes にしたことを、AI コーディングエージェントにする。
Claude Code / Codex / OpenClaw など複数プロバイダのセッションを 1 画面で監視・操作する
ローカルダッシュボード。

- **解決する痛み:** 2026 年の開発者は 3〜10 個のエージェントセッションを並行で走らせるのが
  当たり前になったが、「今どのエージェントが何をしていて、どれが止まっていて、
  どれが承認待ちか」を横断で見る手段がない。ターミナルタブの数だけ認知負荷が増える。
- **なぜバズるか:** 「並行エージェント運用」は 2026 年最大の開発者の関心事
  (agentic infrastructure が CI/CD 並みの基盤になったという言説が主流)。
  ダッシュボード系はスクリーンショット 1 枚・GIF 1 本で価値が伝わるので拡散効率が最強。
  lazygit / k9s / btop が証明している「運用 TUI/ダッシュボードはスターが伸びる」法則に乗る。
- **MVP(2〜4 週):** いまの Saihai を (1) README・UI を英語化、(2) ITB 固有概念を
  「providers」抽象に整理(Claude Code / Codex の 2 アダプタ)、(3) `pipx install` /
  `uvx saihai` 一発起動、(4) ダークテーマのデモ GIF、まで。**新規実装ほぼ不要**。
- **差別化:** Vibe Kanban 等の既存エージェント管理ツールは「タスクを割り当てる」側に寄る。
  Saihai は read-only-safe・typed state 突合・ローカル完結という
  「壊さない管制塔」ポジションを取る。日本語圏発の「采配」というネーミングもフックになる。
- **リスク:** 各 CLI の state ファイル形式変更への追従コスト。アダプタ層を薄く保つこと。

## 2. mcp-doctor — MCP 設定の健康診断ツール

**一言ピッチ:** `brew doctor` の MCP 版。マシン上の全クライアント
(Claude Code / Claude Desktop / Cursor / VS Code / Codex)の MCP 設定を横断スキャンし、
壊れているサーバー・重複・セキュリティリスクを 1 コマンドで診断する。

- **解決する痛み:** MCP サーバーは公開 1 万件超・SDK 月間 9,700 万 DL 規模まで来たが、
  設定はクライアントごとに別ファイルでバラバラ。起動失敗は無言で握り潰され、
  「なぜかツールが出てこない」のデバッグに全員が時間を溶かしている。
- **なぜバズるか:** 対象ユーザー = MCP を使う全開発者(母数が巨大)。
  さらに「ツールポイズニング検出」「env に平文シークレット検出」「未 pin バージョン警告」
  というセキュリティ Lint を载せると、セキュリティ文脈でのシェア(HN・X)が取れる。
  診断ツールは「試すコスト 10 秒」なのでスター転換率が高い。
- **MVP(1〜2 週):** 各クライアントの設定ファイル探索 → サーバー起動ヘルスチェック
  (initialize ハンドシェイク+ tool 列挙)→ 診断レポート(絵文字付きターミナル出力)。
  次段で `--fix`、tool schema のトークンコスト概算、セキュリティルール。
- **差別化:** 公式 Inspector は「単一サーバーの開発デバッガ」。mcp-doctor は
  「マシン全体のフリート監査」で住み分ける。
- **リスク:** クライアント設定形式の追従。ルールをプラグイン化して外部貢献を受ける設計に。

## 3. skillet — Agent Skills のパッケージマネージャ

**一言ピッチ:** `npm install` を Agent Skills(SKILL.md)に。検索・インストール・
バージョン管理・lockfile・共有レジストリを備えた skills 専用パッケージマネージャ。

- **解決する痛み:** Skills はオープン標準化されエコシステムが急拡大中なのに、
  配布手段が「リポジトリを clone して手でコピー」のまま。チーム内での skills 共有、
  更新追従、依存管理が全部手作業。
- **なぜバズるか:** 「新しいエコシステムにはパッケージマネージャが必ず 1 つ生まれ、
  最初に信頼を取ったものが標準になる」— npm / cargo / homebrew の歴史そのもの。
  エコシステムのインフラを取るのは skydoves 型「波に最速で乗る」の最上位互換。
- **MVP(2〜3 週):** `skillet search / add / update / list`。レジストリは最初
  GitHub topic (`agent-skill`) の検索 API + `skillet.lock`。 Claude Code の
  skills ディレクトリ規約に書き込む。次段でチームレジストリ(git リポジトリを
  レジストリとして指定)、署名検証。
- **差別化:** マーケットプレイス系サイトとの違いは「CLI・lockfile・再現性」という
  エンジニア側の作法に寄せること。あなたは skills リポジトリ運用の実体験があるので
  ドッグフーディングが即できる。
- **リスク:** 本家(Anthropic)が公式マネージャを出す可能性。その場合も
  「マルチエージェント(Claude 以外も)対応」の中立レイヤーとして生存可能。

## 4. blackbox — エージェントのフライトレコーダー

**一言ピッチ:** すべてのエージェント実行(プロンプト・ツールコール・diff)を
ローカルに記録し、タイムラインで検索・リプレイ・HTML 共有できる「航空機のブラックボックス」。

- **解決する痛み:** 「先週の火曜、エージェントは俺のリポジトリに何をした?」に
  誰も答えられない。障害調査・コードレビュー・監査・そして「バズった実行例を
  ブログに貼る」需要すべてに対して、実行履歴が使い捨てられている。
- **なぜバズるか:** (1) 監査・コンプライアンス文脈で企業ユーザーが付く、
  (2) 「共有可能な実行リプレイ」は SNS でデモが自己拡散する(asciinema が
  スターを集めたのと同じ構造)、(3) デバッグ用途で毎日開くツールになる。
- **MVP(3〜4 週):** Claude Code の transcript(jsonl)+ hooks を食わせて
  タイムライン UI(ローカル Web)に整形 → セッション検索 → 単一 HTML 書き出し。
  Saihai の「queue/report/evidence の突合」ロジックがほぼ流用できる。
- **差別化:** LangSmith 等の SaaS 観測ツールに対し「ローカル完結・コード不要・
  CLI エージェント特化」。クラウドに送らないことが企業導入の決め手になる。
- **リスク:** transcript 形式の変更追従。プロバイダごとの ingest をプラグイン化。

## 5. agentgate — エージェントのツール実行ファイアウォール

**一言ピッチ:** エージェントの全ツール実行の前段に立つローカルポリシーエンジン。
allowlist / dry-run diff プレビュー / スマホからの承認 / 完全監査ログ。

- **解決する痛み:** エージェントに強い権限を渡すほど生産性は上がるが、
  `rm -rf` 一発・本番 DB 直叩き・シークレット流出が怖い。各ツールの permission 機構は
  プロバイダごとにバラバラで、組織ポリシーとして統一管理できない。
- **なぜバズるか:** エージェントセキュリティは 2025〜2026 年の HN 頻出トピック。
  「firewall for AI agents」という一言で通じるコンセプトの強さがあり、
  企業のセキュリティチームが探しに来るカテゴリ。恐怖訴求はシェアされやすい。
- **MVP(3〜4 週):** Claude Code hooks(PreToolUse)+ Codex 相当のフックに刺さる
  ポリシー評価 CLI。YAML ポリシー(コマンドパターン / パス / ネットワーク)、
  違反時 block + 理由表示、全判定の JSONL 監査ログ。次段でモバイル承認(Push)。
- **差別化:** サンドボックス(コンテナ隔離)系と違い「既存環境に 5 分で入る
  薄いポリシー層」。Saihai の gate / hard-block / typed state の設計思想を
  そのまま単体プロダクト化したものであり、あなたの思想的ホームグラウンド。
- **リスク:** 「ポリシーの書きやすさ」が命。プリセット(recommended.yaml)を同梱し
  ゼロ設定で価値が出るようにする。

## 6. tokenmeter — エージェント横断のローカルコストメーター

**一言ピッチ:** Claude Code も Codex も API 直叩きも、マシン上の全 LLM 消費を
1 つの TUI / ダッシュボードに集約する家計簿。予算アラート付き。

- **解決する痛み:** ツールごとに課金体系もログも別で、「今月いくら使ったか」を
  正確に言える開発者がいない。チームだと把握はさらに絶望的。
- **なぜバズるか:** コスト可視化ツール(例: Claude Code 特化の ccusage)が
  既に大きくバズった実績があり、需要は証明済み。「特化 → 横断」への一般化は
  後発でも取れるポジション。数字とグラフはスクショ映えし、
  「$X 溶かしてた」系ポストと一緒に拡散する。
- **MVP(1〜2 週):** 各ツールのローカルログ/transcript からトークン数を集計し
  料金表を掛けるだけ。日次/プロジェクト別/モデル別のターミナルレポート。
  次段で常駐 TUI、予算超過アラート、チーム集計(git リポジトリに匿名集計を push)。
- **差別化:** 単一ツール特化の既存勢に対し「マルチプロバイダ・マルチエージェント横断」。
  料金表データを別リポジトリ(コミュニティ更新)に分離して貢献導線を作る。
- **リスク:** 料金改定への追従。↑の分離で外部貢献に任せる構造にして解決。

## 7. vaultmind — Markdown ネイティブなエージェント長期記憶

**一言ピッチ:** エージェントの長期記憶を「Obsidian 互換のプレーン Markdown vault」に
する記憶レイヤー。人間が読める・編集できる・git で差分管理できる記憶。

- **解決する痛み:** エージェントの記憶(learnings、決定事項、プロジェクト知識)は
  各ツールの独自形式・不透明 DB・クラウドに分散し、人間が監査も編集もできない。
  「エージェントが何を覚えているか分からない」ことが信頼の最大の障壁。
- **なぜバズるか:** メモリは 2026 年のエージェント開発の最重要未解決問題で、
  かつ「ローカル・プレーンテキスト・自分のデータは自分のもの」という
  local-first 思想は OSS コミュニティの琴線ど真ん中(Obsidian コミュニティが実証済み)。
  この 2 つの波の交点はまだ定番が空席。
- **MVP(3〜4 週):** MCP サーバーとして実装(全エージェントから使える)。
  `remember / recall / forget` ツール + 決まったノート規約(frontmatter で
  scope/confidence/expiry)+ デイリーノートへの追記。既存 vault を壊さない設計。
- **差別化:** ベクタ DB 系メモリ(mem0 等)に対し「人間可読・git 管理・
  Obsidian でそのまま開ける」。あなたは Vault 運用の設計知見(AGENTS_VAULT_ROOT /
  USER_VAULT_ROOT の分離など)を既に持っている。
- **リスク:** 検索品質。まず全文検索 + タグで十分と割り切り、埋め込みはオプションに。

## 8. agentci — エージェント設定の回帰テスト基盤

**一言ピッチ:** CLAUDE.md・skills・プロンプト・MCP 設定を「コード」として扱い、
PR で変更されたらシナリオテストを回す CI ハーネス。「agent config に対する Jest」。

- **解決する痛み:** チームの生産性がエージェント設定の品質に依存する時代になったのに、
  CLAUDE.md の 1 行変更が挙動をどう変えたかを検証する手段がなく、全員が
  「なんとなく良くなった気がする」で運用している。設定リポジトリにテストがない。
- **なぜバズるか:** 「Prompts are code, test them」は必ず来る規範で、
  規範を最初に道具化したツールがカテゴリ名になる(ESLint / Prettier の歴史)。
  対象がエージェント利用チーム全部なので TAM が広い。
- **MVP(3〜4 週):** `agentci.yaml` にシナリオ(入力プロンプト+期待アサーション:
  「このツールを呼ぶこと」「このファイルに触れないこと」等)を書き、ヘッドレスで
  エージェントを実行して判定する GitHub Action + ローカル CLI。LLM-as-judge は
  次段でよい(まず決定的アサーションだけで価値が出る)。
- **差別化:** promptfoo 等の「プロンプト評価」勢は API 呼び出し単位。agentci は
  「エージェント実行(ツールコール列・ファイル変更)単位」のアサーションで住み分ける。
  Saihai の workflow contract 検証(schema + selector + tests)の発想の直系。
- **リスク:** 実行コストと flakiness。デフォルトを安価モデル+リトライ+
  記録リプレイ(blackbox と連携!)で抑える。

## 9. AOC — Agent Orchestration Contracts(型付きワークフロー契約の標準)

**一言ピッチ:** Saihai の P0 workflow contract(schema / template / deterministic
selector)を切り出し、「OpenAPI がやったことをマルチエージェントワークフローにやる」
中立標準 + バリデータ + 可視化ツールとして公開する。

- **解決する痛み:** LangGraph / CrewAI / Dify / 自作 — オーケストレーション実装は
  乱立しているが、「どのロールが・どの入力で・何を成果物として返し・誰がレビューするか」
  という契約層はどこも独自定義で、移植も監査も比較もできない。
- **なぜバズるか:** 標準化プレイはホームランかゼロかの博打だが、当たれば
  スターではなく「エコシステムの座標」が手に入る(MCP 自身が証明した)。
  仕様だけでなく **リファレンス実装(validator)+ 可視化** を同時に出すのが成功条件。
- **MVP(4〜6 週):** 既存の JSON Schema 群(work-order / *-report / activation-envelope
  / typed-classification)を ITB 文脈から独立させて `aoc validate` CLI 化、
  仕様書(SPEC.md)、Mermaid 可視化、LangGraph へのエクススポータ 1 本。
- **差別化:** フレームワークではなく「契約」に限定することで全フレームワークの
  味方になれる。gate / evidence / review chain を一級市民にした契約は他に無い。
- **リスク:** 10 案中で最も難易度が高く、普及活動(発信・議論)の比重が大きい。
  #1 や #10 で注目を集めた後の二の矢として撃つのが現実的。

## 10. team-in-a-box — ワンコマンドで入る「AI 開発チーム一式」(ショーケース戦略)

**一言ピッチ:** `npx team-in-a-box init` で、任意のリポジトリに
「PM・実装・レビュー・セキュリティ・テストのロール一式 + gate 付きワークフロー +
ダッシュボード(Saihai)」が入る、opinionated な AI 開発チームのディストリビューション。

- **解決する痛み:** マルチエージェント運用の記事はあふれているのに、
  「動く完全な形」を 5 分で手に入れる方法がない。全員が自前で組んで消耗している。
- **なぜバズるか:** これは skydoves の **Pokedex 戦略(ショーケースが一番伸びる)**
  のエージェント版。「AI 開発チームをリポジトリに入れる」という体験はデモ動画映えが
  極端に良く、「導入 1 コマンド」は記事化・動画化されやすい。ライブラリ 10 個分の
  集客装置として、#1〜#9 のどれを作る場合でも玄関として機能する。
- **MVP(3〜4 週):** ロール定義(organization/roles の汎用化)+ ワークフロー
  テンプレート(P0 templates の汎用化)+ init CLI + チュートリアル
  (「TODO アプリをこのチームに作らせる」の実録つき README)。
- **差別化:** フレームワークを作らない。Claude Code / Codex の**既存機能の上に載る
  「構成のディストリビューション」**に徹することで、本家アップデートが追い風になる。
- **リスク:** opinionated ゆえの好み分かれ。preset を複数(solo / startup / enterprise)
  用意して回避。

## 推奨の攻め順

1. **まず #1(Saihai 国際化)** — 新規実装がほぼ不要で、既存資産がそのまま
   グローバル市場に出せる。英語 README + GIF + `uvx` 一発起動 + Show HN。
   ここで「エージェント運用ツールの人」という看板を立てる。
2. **並行して #2(mcp-doctor)か #6(tokenmeter)** — 1〜2 週で出せる小粒で
   母数の大きい単発ツール。skydoves 式「小さいリポジトリの量産と相互送客」を開始する。
   README から Saihai へリンクし、ポートフォリオ間でトラフィックを回す。
3. **看板が立ったら #10(team-in-a-box)** — ショーケースで一気にスターを取りに行き、
   その中の部品として #3 / #5 / #9 を順次切り出していく。

## バズらせるための共通プレイブック(全案共通)

- **README = 製品。** 冒頭にバナー画像、10 秒で伝わる GIF、バッジ、
  「インストール 1 行 → 動く」まで 3 スクロール以内。skydoves の README を型として写経する。
- **命名は発音可能な 1 単語**、CLI 名 = リポジトリ名 = 検索キーワード。
- **導入コストを 10 秒に。** `uvx` / `npx` / `brew` で試せない OSS はスターされない。
- **ローンチは束で:** GitHub 公開 → Show HN → X(英語 + 日本語)→ Reddit
  (r/ClaudeAI, r/LocalLLaMA 等)→ Zenn/Qiita 記事。日本語圏で先に火をつけて
  英語圏に飛び火させるのは ccusage 等が実証した勝ちパターン。
- **最初の 48 時間は issue 即応。** 初期ユーザーの issue に 1 時間以内で返すと
  コントリビュータに転化する。
- **`good first issue` を最初から 5 個仕込む。** スターは見る人、コントリビュータは
  賑わいを作る人。改廃が賑わう OSS = 外部 PR が回る OSS。
- **リリースを刻む。** 毎週小さくてもリリースノートを出す。動いているリポジトリだけが
  スターされ続ける。
